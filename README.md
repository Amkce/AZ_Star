# AZ_Star_New 中期汇报说明

## 0. 说明
本文档目前由以下内容构成：

- `star_main.py`
- `form.ui`
- `sky_widget.py`
- `map_widget.py`
- `stellarium_panel.py`
- `stellarium_panel.ui`

当前版本中：

- 真实卫星跟踪逻辑主要写在 `star_main.py` 内。
- `Stellarium` 联动是独立面板，用于联动、同步和回放。

---

## 1. 项目目标
本项目是一个面向赤道仪卫星跟踪的桌面程序，目标不是单纯显示卫星位置，而是把“卫星抓取、轨迹预览、真实跟踪、外部联动”串成一套完整流程。

当前已经实现的核心目标有四类：

1. 从 Heavens-Above 抓取当天可观测卫星及对应 TLE。
2. 在主界面中预览卫星的天空球轨迹和地面轨迹。
3. 通过 ASCOM 控制赤道仪，对卫星执行 `Slew + MoveAxis + 二次 Slew` 的混合跟踪。
4. 通过 `Stellarium Remote Control` 与本机 Stellarium 联动，做可视化回放。

---

## 2. 当前代码结构

### 2.1 主文件
`star_main.py` 是当前项目的核心文件，主要负责：

1. 主界面逻辑。
2. Heavens-Above 抓取。
3. 表格数据填充。
4. 天空球与地面轨迹的计算和刷新。
5. ASCOM 赤道仪连接与真实跟踪。
6. 打开 `Stellarium` 联动窗口。

可以理解为：当前版本的“主控制器”就在这个文件里。

### 2.2 主界面 UI
`form.ui` 定义主界面布局，主要包含：

1. 赤道仪连接区。
2. 经纬度、偏差阈值、时区设置区。
3. 天空球和地面地图显示区。
4. 卫星表格区。
5. 模拟跟踪模式、预备跟踪模式、手动微调区。
6. 搜索设置区。
7. 本地 TLE 输入区。

此外，由于早期不规范，其命名并非是“逻辑_ui”的命名，但其是支撑star_main.py的ui文件。

### 2.3 可视化控件
`sky_widget.py` 和 `map_widget.py` 是两个自绘控件：

1. `sky_widget.py` 负责“地面视角的天空球轨迹”。
2. `map_widget.py` 负责“卫星 subpoint 的地面轨迹”。
3. 这两个文件均是小组件，依赖ui_form.py中的Widget实现。

### 2.4 Stellarium 联动模块
`stellarium_panel.py` 和 `stellarium_panel.ui` 负责 Stellarium 联动窗口。

这个模块现在负责：

1. 展示当前已抓取卫星列表。
2. 同步 TLE 到 Stellarium 的 `satellites.json`。
3. 同步观测地。
4. 同步时间。
5. 定位卫星。
6. 控制 Stellarium 进行普通回放。

注意：当前版本的 `Stellarium` 面板是联动与回放工具，不是主跟踪算法本体。

## 3. 系统整体流程
从用户操作角度看，当前程序的工作流如下：

1. 设置经纬度、时区、偏差阈值（后期可能会做预设设置，目前预设是按测试者的这边设计的）。
2. 连接赤道仪。
3. 从 Heavens-Above 抓取当天可见卫星。
4. 在表格中单击某一行，预览该卫星的天空球轨迹和地面轨迹。
5. 在表格中双击某一行，启动真实卫星跟踪。
6. 如果需要做外部可视化验证，则打开 `Stellarium` 联动窗口。

也可以概括成：

```text
Heavens-Above 抓取
    -> 表格显示
    -> 单击预览轨迹
    -> 双击开始跟踪
    -> 可选：送入 Stellarium 做联动回放
```

## 4. 卫星抓取模块

### 4.1 为什么使用 Heavens-Above
当前项目没有自己维护完整的卫星过境数据库，而是直接利用 Heavens-Above 提供的“某地点当天可见卫星列表”。

这样做的优点是：

1. 直接获得“今天哪些卫星值得看”。
2. 直接获得起始、最高点、结束时间等观测窗口信息。
3. 可以继续进入详情页抓取对应 TLE。

也就是说，本项目并不是先有海量 TLE 再自己筛选，而是先从观测网站拿到“适合今天看的卫星”，再补齐跟踪所需的 TLE。

### 4.2 抓取实现
抓取逻辑在 `star_main.py` 的 `TodaySatelliteScraper` 中。

主要分成两步：

1. 请求 `AllSats.aspx`
   - 获取卫星名称
   - 获取星等
   - 获取起始时间、最高点时间、结束时间
   - 获取最大高度
   - 获取卫星 ID

2. 请求 `orbit.aspx`
   - 针对每颗卫星补抓 `TLE1` 和 `TLE2`

### 4.3 当前抓取参数设计
当前抓取参数有几个关键点：

1. 地点来自主界面的经纬度。
2. `loc` 使用 `Observer`，（当然，这个没有实际意义）。
3. 抓取时区使用 `ChST`，以确保时区正确。
4. 语言优先使用中文请求头。
5. 支持“凌晨模式 / 傍晚模式”筛选。
6. 支持“最低亮度”筛选。

这部分设计的目的，是尽量让抓到的结果更贴近本地实际观测习惯，而不是只拿到一份原始英文列表。

### 4.4 时间处理策略
抓取回来的时间文本本身只是类似 `18:12:46` 这样的本地时刻字符串，因此程序还需要把它恢复成可计算的 UTC 时间。

当前做法是：

1. 从链接参数里提取 `mjd` 作为时间锚点。
2. 用锚点先恢复出一个 UTC 基准时间。
3. 再结合源站时区，把 `HH:MM:SS` 解析成接近该锚点的完整 UTC 时间。

这样做的原因是：

1. 表格里后续要做窗口期判断。
2. 天空球轨迹要按该行窗口时间精确绘制。
3. 跟踪模式下要判断“当前是否处于窗口期”。

### 4.5 为什么保留本地 TLE 输入
主界面仍然提供本地 TLE 输入框，主要是因为以下原因：

1. 有些条目能抓到过境信息，但拿不到 TLE。
2. 某些调试场景下，固定 TLE 比在线抓取更稳定。
3. 用户可能本来就已经拥有目标卫星的 TLE。

所以当前程序支持两种入口：

1. 表格条目跟踪。
2. 手动粘贴本地 TLE 后直接跟踪。

## 5. 轨迹预览模块
当前程序有两类轨迹预览：

1. 天空球轨迹。
2. 地面轨迹。

### 5.1 天空球轨迹
天空球轨迹由 `NextPassWorker` 计算，绘制由 `SkyTrackWidget` 完成。

它的语义是：

“站在观测地抬头看天空，这颗卫星在本次窗口中的运动路径。”

当前实现里有两个重要特点：

1. 如果用户单击了表格中的某一行，那么天空球按该行的窗口起止时间绘制。
2. 如果没有绑定窗口，则退回到“未来 48 小时内下一次可见过境”。

这比简单画“下一次过境”更符合实际使用场景，因为主界面表格本来就代表了一次具体的观测窗口。

### 5.2 天空球绘制规则
`sky_widget.py` 当前做了这些可视化设计：

1. 以圆盘表示天空。
2. 外圆表示地平线。
3. 同心圆表示不同高度角。
4. 方位语义采用“左东右西”。
5. 起点、终点和最高点会标出来。
6. 每隔大约 2 分钟做一次小字体时间标记。
7. 时间标记旁边带一个小点，便于辨认。

这样用户不仅能看出轨迹形状，还能直观看出什么时候经过哪个位置。

### 5.3 地面轨迹
地面轨迹由 `GroundTrackWorker` 计算，绘制由 `MapTrackWidget` 完成。

它的意义是：

“卫星在未来一段时间内投影到地球表面的轨迹。”

当前实现中：

1. 计算未来 120 分钟的轨迹。
2. 采样步长默认 10 秒。
3. 第一个点视为当前点，会用特殊标记显示。

### 5.4 视觉翻转
用户界面里有 TLE 翻转复选框，当前不仅影响主跟踪方向，也影响天空球和地图的显示。

具体来说：

1. 左右翻转会改变天空球方位映射，并反转地图经度。
2. 上下翻转会改变天空球方位映射，并反转地图纬度。

这样做的目的，是给现场校准留出空间，因为不同设备方向定义可能不完全一致。

## 6. 真实跟踪模块
这是当前项目最核心的技术部分。

### 6.1 总体思路
项目没有采用“只用 Slew”或者“只用 MoveAxis”的单一方案，而是采用混合策略：

1. 启动前先做一次 `Slew`，把赤道仪快速送到目标附近。
2. 进入跟踪后，以 `MoveAxis` 连续微调，实现平滑跟踪。
3. 如果发现当前指向与目标偏差过大，则执行二次 `Slew` 重新捕获。

这个方案背后的考虑是：

1. 只用 `Slew`，动作不连续。
2. 只用 `MoveAxis`，初始偏差大时不容易追上。
3. 混合方案更符合真实设备控制需求。

### 6.2 跟踪启动入口
真实跟踪的入口是 `star_main.py` 里的 `_start_tracking_with_tle()`。

它会依次做这些事：

1. 检查赤道仪是否已连接。
2. 检查经纬度是否已设置。
3. 用 `Skyfield` 构造卫星对象和观测者对象。
4. 检查设备是否驻车，如驻车则尝试解驻车。
5. 检查 `MoveAxis` 能力是否可用。
6. 检查 `SlewToCoordinates` 是否可用。
7. 根据当前模式决定起始时间。
8. 启动前先执行一次 `Slew`。
9. 启动周期性 `tick`。

### 6.3 三种时间模式
当前跟踪逻辑支持三种时间语义：

1. 实时模式
   - 未勾选“模拟追踪模式”时使用。
   - 按真实当前 UTC 运行。
   - 仅允许在窗口期内开始跟踪。

2. 模拟模式
   - 勾选“模拟追踪模式”时使用。
   - 程序内部维护一条虚拟时间轴。
   - 虚拟时间通常从下一次 AOS 开始推进。

3. 预备跟踪模式
   - 在非模拟模式下勾选“预备跟踪模式”时使用。
   - 程序先 `Slew` 到下一次升起点附近。
   - 在窗口未开始时保持等待。
   - 等到 AOS 到来后切换为正式跟踪。

### 6.4 初始 Slew 如何实现
初始 `Slew` 的做法不是“随便找个初始角度”，而是：

1. 选定一个目标时刻。
2. 用 `Skyfield` 根据 TLE、观测地和该时刻计算目标 `RA/Dec`。
3. 调用 `SlewToCoordinates` 或 `SlewToCoordinatesAsync`。

这里的目标时刻在不同模式下不同：

1. 实时模式：当前时刻。
2. 模拟模式：虚拟跟踪时刻。
3. 预备模式：下一次 AOS 时刻。

### 6.5 为什么要区分赤道坐标系
当前代码专门读取了驱动的 `EquatorialSystem`，原因是不同 ASCOM 驱动使用的赤道坐标系可能不同。

当前做法是：

1. 如果驱动使用 `J2000`，则按 J2000 计算。
2. 否则按 `epoch='date'` 的当日历元坐标计算。

这么做的原因是：如果目标 `RA/Dec` 坐标系与驱动读写的坐标系不一致，`Slew` 看起来就会“不准”甚至像“没起作用”。

### 6.6 周期性跟踪 tick
真实跟踪由 `QTimer` 周期触发 `_tracking_tick()`。

当前每个 tick 主要做这些事情：

1. 计算当前应跟踪到的目标时刻。
2. 用 `Skyfield` 计算该时刻卫星的 `Alt/Az`。
3. 计算该时刻卫星的目标 `RA/Dec`。
4. 读取赤道仪当前 `RA/Dec`。
5. 计算当前指向与目标指向的球面角距离误差。
6. 如果误差超过阈值，则触发二次 `Slew`。
7. 如果误差未超阈值，则根据前后两次采样差分估算双轴速度。
8. 叠加用户的手动微调速度。
9. 调用 `MoveAxis` 发送最终轴速率。
10. 更新界面标签。

### 6.7 连续跟踪速度为什么用差分
当前程序直接用相邻采样点做差分：

```text
轴速率 ≈ (当前目标坐标 - 上一次目标坐标) / dt
```

这样做的优点是：

1. 实现简单。
2. 对不同卫星、不同过境角度都适用。
3. 不依赖人为推导某一种固定轨道模型的控制公式。

代价是它会依赖采样周期，因此程序需要通过较高频率定时刷新来保持连续性。

### 6.8 二次 Slew 的作用
二次 `Slew` 不是异常补救，而是控制策略的一部分。

它主要解决这些问题：

1. 初始对位仍然有残余误差。
2. 赤道仪响应存在迟滞。
3. 手动微调可能引入偏差。
4. 高仰角区域目标运动快，误差更容易放大。

当前触发条件大致是：

1. 当前误差大于设定的偏差阈值。
2. 当前不在 `Slew` 稳定保护期内。
3. 距离上一次二次 `Slew` 已超过最短间隔。

触发时会：

1. 按一定前瞻时间重新计算目标位置。
2. 做一次新的 `Slew`。
3. 清空前一轮差分缓存。
4. 重新进入连续跟踪。

### 6.9 手动微调如何工作
主界面的方向键和速度滑条不是独立模式，而是叠加在自动跟踪之上的辅助输入。

当前逻辑是：

1. 自动跟踪速度来自卫星运动差分。
2. 手动微调速度来自方向键和速度滑条。
3. 最终发送给 `MoveAxis` 的是两者叠加后的速度。

这样做的好处是：

1. 用户可以在跟踪过程中做小修正。
2. 松开方向键后自动跟踪不会停止，只会回到纯自动速率。

### 6.10 速度下发安全性
当前项目不会盲目把任意速度发给 `MoveAxis`，而是会先查询驱动的 `AxisRates`。

后续会做这些保护：

1. 识别每一轴允许的速度区间。
2. 过小速度可能归零。
3. 不在合法区间时，会吸附到最近合法边界。
4. 超过最大允许速度时会被钳制。

这样可以避免下发驱动不接受的速率。

## 8. Stellarium 联动模块

### 8.1 模块定位
当前 `Stellarium` 模块的定位是：

“把主程序抓到的卫星结果送进 Stellarium，用于联动观察和回放。”

它不是主跟踪算法本体，也不是赤道仪控制器。

### 8.2 当前界面结构
`stellarium_panel.ui` 当前是左右分栏：

1. 左侧显示“当前已抓取卫星”列表。
2. 右侧显示当前选中卫星的详细信息和控制按钮。

右侧当前可做的操作包括：

1. 启动 Stellarium。
2. 检查连接状态。
3. 一键联动。
4. 写入/更新卫星。
5. 同步观测地。
6. 跳到起始时刻。
7. 定位到卫星。
8. 按指定倍率开始回放。
9. 暂停回放。
10. 恢复实时。

### 8.3 和 Stellarium 的通信方式
当前采用 Stellarium 的 `Remote Control` 插件提供的 HTTP 接口。

`stellarium_panel.py` 中封装了 `StellariumClient`，主要负责：

1. 读取状态。
2. 读取插件状态。
3. 设置地点。
4. 设置时间。
5. 设置时间倍率。
6. 搜索对象。
7. 聚焦对象。
8. 执行动作命令。

### 8.4 为什么要写 satellites.json
仅仅通过远程接口并不能保证一颗新卫星在 Stellarium 内可被搜索到，因此当前程序还会直接写入用户配置目录下的 `satellites.json`。

这样做的好处是：

1. 把当前抓到的 TLE 真正导入到 Stellarium 的卫星系统里。
2. 后续可以按卫星名称或 NORAD ID 搜索。
3. 如果目标原本不存在，也能补进去。

因此当前联动模块本质上分两部分：

1. 文件层同步卫星数据。
2. HTTP 层控制 Stellarium 界面行为。

### 8.5 一键联动的语义
当前“一键联动”并不是神秘黑箱，而是几步操作的组合：

1. 检查 Stellarium 是否在线。
2. 把当前卫星写入或更新到 `satellites.json`。
3. 同步当前观测地。
4. 把 Stellarium 时间跳到这颗卫星的起始时刻。
5. 尝试在 Stellarium 中找到并定位该卫星。

之后用户还可以继续点“开始模拟”，让 Stellarium 按给定倍率播放。

## 9. 设计理念总结
如果把当前项目的实现理念压缩成几句话，我认为主要有以下几点。

### 9.1 先解决“能不能用”，再谈“够不够漂亮”
很多实现都明显偏工程实用主义，例如：

1. 直接抓 Heavens-Above，而不是自建完整数据库。
2. 直接用 TLE 和 Skyfield 做实时计算，而不是另造轨道引擎。
3. 直接读取驱动的 `AxisRates` 做兼容，而不是假定所有驱动都一样。

这些选择的重点不是形式优雅，而是“尽快形成完整可用链路”。

### 9.2 真实控制和可视化验证并重
这个项目并没有只做设备控制，也没有只做显示界面，而是把两者都做了：

1. 主界面负责真实控制。
2. 天空球和地图负责快速预判。
3. Stellarium 面板负责外部联动验证。

这使得项目既能用于现场操作，也能用于调试和讲解。

### 9.3 尽量让状态可见
当前代码里有大量标签更新、状态提示、窗口期判断、失败报错。
包括：

1. 当前是什么时间基准。
2. 当前是不是在窗口期。
3. 当前发了什么轴速率。
4. 当前是不是在等待 AOS。
5. 当前是不是刚做了二次 Slew。

若缺少这些信息，那现场调试会非常痛苦。

### 9.4 给人工干预留空间
当前系统虽然有自动跟踪，但仍然保留：

1. 手动速度滑条。
2. 方向键微调。
3. TLE 翻转。
4. 本地 TLE 输入。

这说明项目并不是假设“算法一定完美”，而是承认现场设备、安装姿态、目标条件可能复杂，因此需要人为介入能力。

## 10. 当前版本的边界

当前版本仍有这些结构性特征：

1. 主跟踪逻辑集中在 `star_main.py`，文件较大。
2. `Stellarium` 面板目前是联动与回放模块，不是共享跟踪核心。
3. 抓取结果依赖 Heavens-Above 页面结构，页面变动会带来适配成本。
4. Stellarium 新卫星写入后，某些情况下仍可能需要重启 Stellarium 才能完全生效。

这些不是“程序失败”，而是当前工程阶段的自然状态。
